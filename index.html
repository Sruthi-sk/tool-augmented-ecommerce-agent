PartSelect · Chat Agent Case Study
Finding the right appliance part is a precision problem
An AI-powered chat assistant for refrigerator and dishwasher parts — built on a two-service architecture with deterministic grounding and live data fallback.
Next.js + FastAPI LLM orchestration RAG + live fallback Deterministic grounding
Case Study by Sruthi Kuriakose
The problem
Customers describe symptoms. Parts are indexed by keys, model numbers, and SKUs.
Wrong part = return, lost trust, support cost. Every answer must be grounded in real data.
The gap
"My dishwasher makes a grinding noise" must map to symptom_key='noisy', then to compatible parts for an exact model number — approximate answers cause failed repairs
Scope enforcement
Refrigerator and dishwasher parts only. Out-of-scope queries are rejected before any LLM call with a constructive redirect — zero tokens wasted
Ground answers in real data
All facts come from a pre-indexed SQLite store or live PartSelect scraping — never from LLM memory. Every response includes a source URL tracing back to a real product page
Precision over fluency
A deterministic post-generation validator rewrites any response where the stated part number, price, compatibility claim, or URL doesn't match the tool result
Three layers of correctness: data quality at ingestion → LLM instructed not to speculate → deterministic post-generation rewrite if facts don't match.

Stages 1 & 5
Regex + keyword scope check before any LLM call — enforces refrigerator/dishwasher constraint, extracts part and model numbers into session state. Post-generation: validator rewrites hallucinated facts; _append_missing_links() ensures URLs are never dropped.
Stages 2 & 4
Pass 1 selects a tool via function calling — no user-facing text. Pass 2 composes natural language from a structured dict with source_url. Two-pass design separates routing failure from prose failure — independently debuggable. Up to 3 tool calls per turn, enabling chains like search → compatibility in one message.
Stage 3
6 tools, registered via @registry.register() — schemas auto-generate for OpenAI and Anthropic. All tools go through KnowledgeService (SQLite + FAISS). Sub-10ms execution. Tools never touch storage directly — underlying store is swappable without touching tool code.

Data layer
SQLite + FAISS semantic routing + live compatibility fallback
Structured facts from the offline index. Semantic routing for natural language symptoms. Real answers for models not yet cached.
SQLite (offline index)
Crawl4AI (Playwright) ingests category → brand → sub-category → part detail pages. 16+ fields per part, up to 500 compatible models, symptoms + install narratives. Every record has a verified source_url. FTS5 with Porter stemming for keyword search.
FAISS (routing only)
Repair guide chunks embedded via text-embedding-3-small at ingestion. At query time, maps "my fridge makes a grinding noise" → symptom_key via cosine similarity. Adaptive over-fetch guarantees k results. Not a truth source — structured store owns facts.
Live fallback
For models/parts not in the local index: headless Chromium injects JS into PartSelect's compatibility widget. Only definitive results cached. Graceful degradation on 403 — user gets a direct PartSelect link instead of a false negative.
Compatibility resolution chain
Local SQLite hit→ instant, else Live browser check→ Cache to SQLite→ instant next time. On 403 → Direct PartSelect link
Compatibility uses three states: True, False, None. None triggers a live check — never a false negative. PartSelect blocks plain HTTP clients (403), so scraping at query time runs via Playwright; the offline index moves that cost to ingestion.

Extensibility & scalability
Minimal changes for new tools, providers, and appliances
Every abstraction has an explicit upgrade path for production scale.
New tool
@registry.register() — schemas auto-generate for OpenAI and Anthropic. Orchestrator picks it up automatically. Add one case to DetailPanel.tsx.
New LLM provider
Implement LLMProvider ABC → set LLM_PROVIDER=yourprovider. Tool schemas auto-translate per provider. All tools unchanged.
New appliance type
Add keywords to preprocessor.py, category URL to CATEGORY_URLS, repair index URL to REPAIR_INDEXES. appliance_type threads through the whole stack automatically.
  • Persistent sessions — swap InMemorySessionStore with Redis via the same SessionStore ABC. Zero orchestrator changes.
  • Storage at scale — SQLite is read-only at runtime, replaceable with Postgres through the same KnowledgeService interface. FTS5 scales to ~100K rows, then Postgres tsvector + GIN.
  • Vector search at scale — FAISS flat index is O(n), fine for current symptom count. Swap to IVF-PQ or a dedicated ANN service at scale via the same interface.
  • Horizontal scaling — stateless request handling via session_id header. Supports multi-instance deployments with a persistent session store.
